# Design Notes

De-Time docs should feel like a scientific toolkit, not a review dossier.

## Voice

- Lead with user actions: install, run, choose, compare, reproduce.
- Prefer input-output explanations over abstract positioning.
- Keep reviewer evidence available, but do not make it the first user path.
- Avoid defensive copy such as "what this does not claim" unless a boundary is
  needed for correctness.

## Information Architecture

- The top bar stays compact: install, quickstart, methods, tutorials, API, and
  project material.
- Short pages should not appear as primary navigation items unless they answer
  a first-session user question.
- Evidence appendices are linked from their parent page rather than promoted as
  first-class routes.

## Visual System

- Use De-Time's blue/teal scientific-toolkit palette.
- Use cards for choice and comparison, not long uninterrupted bullet lists.
- Use compact pipeline diagrams for input-output workflows.
- Use collapsible FAQ blocks for troubleshooting and release-check detail.
- Keep typography spacious enough for scanning: short paragraphs, fewer dense
  tables on landing pages, and tables reserved for reference pages.
